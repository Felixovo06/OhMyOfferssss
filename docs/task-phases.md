# AI 面试抽题网站任务阶段划分

## 执行原则

本任务书按业务闭环推进，不按前端和后端拆成两条独立主线。每个阶段都需要同时考虑：

- 数据模型
- 服务端能力
- 接口契约
- 前端页面与交互
- AI 能力
- 权限与异常处理
- 端到端验收

涉及 UI 结构、视觉设计、交互体验、组件状态、响应式布局或可用性评审时，必须使用 `ui-ux-pro-max`。包括但不限于：

- 新页面设计
- 页面重构
- 表单、表格、弹窗、导航、卡片等组件实现
- 加载态、空状态、错误态、禁用态
- 移动端适配
- UI 质量检查和可访问性检查

纯后端逻辑、数据库、接口、AI prompt、飞书 API、Redis、部署等任务不需要使用 `ui-ux-pro-max`。

## 阶段 0：项目初始化

目标：搭好可开发、可运行、可连接基础服务的项目骨架。

主要任务：

- 初始化 Next.js App Router + TypeScript 项目。
- 接入 Tailwind CSS 和 shadcn/ui。
- 配置 ESLint、Prettier、基础 tsconfig。
- 接入 Prisma。
- 配置 PostgreSQL 连接。
- 配置 Redis 连接。
- 建立 `.env.example` 和 env 校验。
- 建立基础路由、应用布局和导航。
- 使用 `ui-ux-pro-max` 确认基础后台布局、导航结构和通用状态组件。

关键产出：

- 本地开发服务可启动。
- 数据库连接可用。
- Redis 连接可用。
- 基础页面可访问。
- 基础 UI 框架可复用。

验收标准：

- `npm run dev` 可以启动应用。
- `npm run lint` 通过。
- Prisma 可以连接目标 PostgreSQL。
- Redis ping 成功。
- 后台基础布局在桌面端可正常使用。

## 阶段 1：账号、小组与基础后台

目标：完成内部使用所需的基础登录、小组邀请和后台框架。

主要任务：

- 实现简单邮箱密码登录。
- 创建 `users` 表。
- 创建小组相关表：
  - `groups`
  - `group_members`
  - `group_invitations`
- 实现登录、退出、会话保护。
- 实现工作台首页。
- 实现创建小组。
- 实现邀请链接或邮箱邀请记录。
- 实现接受邀请加入小组。
- 实现当前用户、小组列表、小组成员查询能力。
- 使用 `ui-ux-pro-max` 设计/评审登录页、工作台、小组创建、邀请加入和成员列表体验。

关键产出：

- 登录后才能访问后台。
- 用户可以创建小组并邀请朋友加入。
- 被邀请用户可以加入小组。
- 用户能区分当前使用个人空间还是小组空间。

验收标准：

- 未登录访问后台会跳转登录页。
- 登录成功后可以访问受保护资源。
- 退出后会话失效。
- 小组 owner 可以生成邀请。
- 被邀请用户登录后可以加入小组。
- 小组成员列表能正确展示。

## 阶段 2：题库与题目管理闭环

目标：完成题库和题目的核心维护能力，让用户能手动维护个人题库和小组题库。

主要任务：

- 设计并迁移核心表：
  - `question_banks`
  - `questions`
  - `tags`
  - `question_tags`
- 实现题库 CRUD。
- 实现题目 CRUD。
- 支持题目标签和难度分数。
- 支持按题库、标签、难度、关键词筛选。
- 支持启用/禁用题目。
- 支持题库归属个人或小组。
- 小组成员可以看到小组题库。
- 只有小组 owner 可以管理小组题库。
- 实现题库列表、题库详情、题目编辑、筛选和搜索体验。
- 使用 `ui-ux-pro-max` 设计/评审题库列表、题目表格、编辑表单、筛选器、空状态和删除确认。

关键产出：

- 可以手动创建题库。
- 可以创建小组共享题库。
- 可以手动添加题目、答案、标签和难度分数。
- 可以搜索和筛选题目。

验收标准：

- 创建题库后能在列表中看到。
- 新增题目后能关联到题库。
- 题目必须保存 `difficulty_score` 和标签。
- 禁用题目不会进入后续抽题候选集。
- 小组成员可以使用同一题库进行后续面试。
- 非小组成员不能访问该小组题库。

## 阶段 3：飞书导入闭环

目标：打通飞书文档到导入预览的链路。

主要任务：

- 实现 Feishu Client。
- 支持获取并缓存 `tenant_access_token`。
- 实现飞书文档链接解析。
- 实现文档 blocks 分页拉取。
- 实现 block 到 normalized text 的转换。
- 创建导入相关表：
  - `feishu_sources`
  - `import_batches`
  - `import_items`
- 实现飞书导入任务创建、状态查询和结果查询能力。
- 支持将飞书导入结果归属到个人题库或小组题库。
- 实现飞书导入页、导入状态和错误提示。
- 使用 `ui-ux-pro-max` 设计/评审导入表单、进度状态、错误状态和来源信息展示。

关键产出：

- 输入飞书文档链接后，可以拉取文档内容。
- 系统保存原始 blocks 和归一化文本。
- 导入批次状态可追踪。

验收标准：

- 有权限的飞书文档可以成功读取。
- 无权限、链接错误、限流等场景有明确错误提示。
- 同一导入批次能追溯飞书来源 URL 和 document ID。
- 用户只能向自己有管理权限的题库导入内容。

## 阶段 4：AI 抽取题目与导入确认闭环

目标：让 AI 从飞书内容中抽题，并由用户确认后入正式题库。

主要任务：

- 实现统一 LLM Client。
- 配置 `deepseek4flash`。
- 默认关闭思考模式。
- 编写飞书内容抽题 prompt。
- 定义 AI 抽题 JSON Schema。
- 实现 AI 输出校验。
- 将 AI 抽取结果写入 `import_items`。
- 支持编辑待导入题目的问题、答案、标签、难度分数。
- 支持删除不需要的待导入题。
- 支持批量确认入库。
- 实现导入确认页和来源片段查看。
- 使用 `ui-ux-pro-max` 设计/评审导入确认表格、低置信度提示、批量操作和编辑体验。

关键产出：

- 飞书文档内容可以被 AI 转成结构化题目。
- AI 抽取结果可以在入库前被检查和修改。
- 确认后题目进入正式题库。

验收标准：

- 每道入库题都有问题、标签、难度分数和来源信息。
- AI 低置信度结果有提示。
- 批量入库使用事务，失败时整体回滚。
- AI JSON 解析失败时有重试或失败提示。

## 阶段 5：普通面试闭环

目标：实现普通面试能力，由大模型基于题库、标签、难度和历史表现抽题。

主要任务：

- 创建练习相关表：
  - `practice_sessions`
  - `practice_session_questions`
- 实现面试会话创建能力。
- 支持选择面试模式：普通面试。
- 支持选择个人题库或小组题库。
- 支持选择题量、标签、难度倾向和练习目标。
- 实现候选题筛选。
- 将候选题摘要和历史表现传给大模型。
- 实现普通面试抽题 prompt。
- 校验大模型返回的题目 ID。
- 保存抽题理由和整体策略。
- 实现规则兜底抽题。
- 实现普通面试配置页和抽题结果展示。
- 使用 `ui-ux-pro-max` 设计/评审面试配置、模式选择、抽题结果、无题目状态和开始面试流程。

关键产出：

- 用户可以创建一轮普通面试。
- 本轮题目由大模型选择。
- 每题保存抽题理由。
- 小组成员可以基于同一个小组题库生成各自的练习会话。

验收标准：

- 抽题结果只包含启用且符合筛选条件的题目。
- 每道被抽中的题都有 reason。
- 大模型失败时系统仍能生成练习会话。
- 最近练过的题可以被降权。
- 用户不能从无权限的小组题库抽题。

## 阶段 6：客制化面试闭环

目标：实现基于 PDF 简历的客制化面试，由大模型结合简历和题库抽题并生成追问方向。

主要任务：

- 创建简历相关表：
  - `resumes`
- 支持 PDF 简历上传。
- 优先直接解析 PDF 文本。
- 判断扫描件或图片型 PDF，并提示需要 OCR 或文本版 PDF。
- 实现简历结构化摘要。
- 支持选择面试模式：客制化面试。
- 客制化面试将简历摘要、候选题摘要和历史表现传给大模型。
- 实现客制化面试抽题 prompt。
- 保存抽题理由和与简历相关的追问方向。
- 实现简历上传、解析状态、简历摘要和客制化面试配置体验。
- 使用 `ui-ux-pro-max` 设计/评审简历上传、解析状态、扫描件提示、简历摘要展示和客制化面试配置。

关键产出：

- 用户可以上传文本型 PDF 简历。
- 系统可以解析简历文本并生成结构化摘要。
- 客制化面试题目由大模型结合简历摘要和题库候选集选择。
- 客制化面试保存与简历相关的追问方向。

验收标准：

- 文本型 PDF 简历可以解析出文本和结构化摘要。
- 扫描件 PDF 有明确提示，不静默失败。
- 客制化面试抽题理由会体现简历技能、项目或经历。
- 追问方向不能凭空编造简历中不存在的经历。

## 阶段 7：AI 面试练习与总结闭环

目标：完成一次从抽题、回答、评分、追问到总结的完整闭环。

主要任务：

- 创建 `practice_answers` 表。
- 实现练习会话读取和推进能力。
- 支持普通面试和客制化面试两种练习上下文。
- 支持逐题回答。
- 调用 AI 对回答评分。
- 输出亮点、缺失点、参考表达和追问。
- 客制化面试追问需要参考简历内容、题库问题或用户回答。
- 支持进入下一题。
- 实现会话总结。
- 统计薄弱标签和推荐复习题。
- 实现面试练习页、单题反馈和练习总结页。
- 使用 `ui-ux-pro-max` 设计/评审答题体验、反馈面板、追问展示、进度展示和总结页。

关键产出：

- 用户可以完成一轮 AI 面试练习。
- 每题回答和 AI 反馈被保存。
- 客制化面试可以生成和简历相关的追问。
- 会话结束后生成总结。

验收标准：

- 用户回答后能得到 0 到 100 分评分。
- 单题反馈包含缺失点和改进表达。
- 客制化面试的追问不能凭空编造简历中不存在的经历。
- 会话总结包含总体评分、薄弱标签和下一轮建议。
- 历史练习记录可被后续抽题使用。
- 小组题库下的练习记录归属具体用户，不覆盖其他成员记录。

## 阶段 8：上线前整理

目标：完成 MVP 上线前的稳定性、可维护性和 UI 质量检查。

主要任务：

- 补充关键错误日志。
- 隐藏敏感信息。
- 增加基础种子数据。
- 编写部署说明。
- 编写飞书应用配置说明。
- 完成一次端到端回归测试。
- 使用 `ui-ux-pro-max` 做上线前 UI/UX 质量评审，覆盖可访问性、表单反馈、加载态、空状态、错误态和响应式布局。

关键产出：

- MVP 可以部署给内部用户使用。
- 新用户知道如何配置飞书和环境变量。
- 核心页面 UI/UX 达到可用标准。

验收标准：

- 全链路可跑通：登录 -> 创建小组 -> 飞书导入 -> AI 抽题入库 -> 普通面试 -> 回答评分 -> 总结。
- 客制化链路可跑通：上传简历 -> 解析简历 -> 客制化抽题 -> 回答评分 -> 总结。
- `.env` 不会被提交。
- 错误信息对用户可理解。
- README 或部署文档可指导启动项目。

## 推荐执行顺序

```text
阶段 0 项目初始化
-> 阶段 1 账号、小组与基础后台
-> 阶段 2 题库与题目管理闭环
-> 阶段 3 飞书导入闭环
-> 阶段 4 AI 抽取题目与导入确认闭环
-> 阶段 5 普通面试闭环
-> 阶段 6 客制化面试闭环
-> 阶段 7 AI 面试练习与总结闭环
-> 阶段 8 上线前整理
```

## 最小可用闭环

如果想最快看到效果，可以把第一轮目标压缩为：

1. 阶段 0：项目能运行。
2. 阶段 1：能登录并创建小组。
3. 阶段 2：能手动建题库和题目。
4. 阶段 3：能从飞书拉内容。
5. 阶段 4：AI 能抽题并确认入库。
6. 阶段 5：普通面试能由大模型抽出一组题。

完成这些后，即使客制化面试、AI 评分和总结还没完全做完，也已经具备核心价值。

# AI 面试抽题网站 PRD

## 1. 背景

初期用户是产品创建者及朋友，主要面向面试准备场景。现有面试题沉淀在飞书文档中，因此第一版必须支持飞书 API 导入，并通过 AI 从飞书文档内容中抽取题目、参考答案、标签和难度分数。

产品第一阶段不是完整公开平台，而是一个小范围高频使用的内部面试练习工具：能把飞书题库稳定同步进系统，能按目标岗位和练习状态由大模型智能抽题，能记录练习表现，并逐步沉淀个人薄弱点。

## 2. 产品目标

### 2.1 第一版目标

- 支持通过飞书文档链接导入题库内容。
- 使用 AI 从飞书文档内容中结构化抽取题目和答案。
- 每道题入库时保存标签、难度分数、来源信息。
- 支持用户手动新增、编辑、删除题目。
- 支持由大模型根据用户目标、题库范围、标签、难度和历史表现智能抽题。
- 支持基础 AI 面试练习和练习记录。

### 2.2 非目标

- 第一版不做公开题库市场。
- 第一版不做复杂爬虫系统。
- 第一版不做多人组织权限体系。
- 第一版不强依赖语音面试，先以文字面试为主。
- 第一版不追求完全自动入库，AI 抽取结果需要人工确认。

## 3. 用户角色

### 3.1 题库维护者

通常是产品创建者本人，负责接入飞书文档、确认 AI 抽取结果、维护题目质量。

核心诉求：

- 快速把飞书里的题导入系统。
- AI 能尽量准确地区分问题、答案、说明和代码片段。
- 导入前可以批量检查、修改和丢弃低质量结果。

### 3.2 面试练习者

初期是产品创建者及朋友，用于准备面试。

核心诉求：

- 不想手动挑题，希望系统按当前状态智能抽题。
- 希望题目覆盖全面，不只随机重复简单题。
- 希望回答后获得反馈，知道哪里没答好。

## 4. 核心用户流程

### 4.1 飞书导入流程

1. 用户进入「飞书导入」页面。
2. 输入飞书文档链接。
3. 系统解析文档 ID。
4. 后端使用飞书 API 拉取文档 block 内容。
5. 系统将 block 内容转换为结构化文本。
6. AI 识别并抽取题目、参考答案、标签、难度分数。
7. 系统展示「导入预览」。
8. 用户批量确认、编辑或删除待导入题目。
9. 用户提交后，题目进入数据库。

### 4.2 手动维护流程

1. 用户进入题库详情页。
2. 点击新增题目。
3. 填写问题、参考答案、标签、难度分数。
4. 保存后进入题库。
5. 后续可编辑题目内容、标签、难度和启用状态。

### 4.3 大模型抽题流程

1. 用户进入「开始练习」页面。
2. 选择练习目标，例如：
   - 前端面试
   - 后端面试
   - 项目经历
   - 算法基础
   - 高频八股
   - 查漏补缺
3. 用户选择题库范围、题量、难度倾向。
4. 系统将候选题、用户历史表现、目标和约束传给大模型。
5. 大模型返回本轮题目组合及抽题理由。
6. 系统创建练习会话。
7. 用户逐题回答。
8. AI 对回答给出反馈和评分。
9. 系统记录本次练习结果。

## 5. 功能需求

## 5.1 飞书 API 接入

### 5.1.1 飞书应用配置

系统需要支持配置：

- `FEISHU_APP_ID`
- `FEISHU_APP_SECRET`
- 飞书 API 基础地址
- token 缓存过期时间

第一版建议使用飞书企业自建应用，通过 `tenant_access_token` 调用云文档 API。

### 5.1.2 文档链接解析

系统需要从飞书文档 URL 中解析：

- 文档类型
- `document_id`
- 原始 URL

第一版优先支持新版飞书文档。其他类型如多维表格、知识库空间、老版文档可后续扩展。

### 5.1.3 文档内容读取

系统通过飞书文档 block API 拉取文档内容。

要求：

- 支持分页读取。
- 支持 token 缓存。
- 支持基础错误提示，例如无权限、文档不存在、接口限流。
- 保存导入批次的原始结构化内容，方便排查 AI 抽取问题。

### 5.1.4 Block 转文本

系统需要将飞书 block 转为适合大模型理解的中间格式。

建议格式：

```text
# 标题

## 二级标题

Q: 问题文本
A: 答案文本

代码块:
...

列表:
- ...
```

转换时需要保留：

- 标题层级
- 普通文本
- 有序/无序列表
- 代码块
- 表格文本
- block ID

## 5.2 AI 导入抽取

### 5.2.1 抽取目标

AI 需要从飞书内容中抽取结构化题目：

```json
{
  "question": "什么是浏览器事件循环？",
  "answer": "参考答案文本",
  "tags": ["JavaScript", "浏览器", "事件循环"],
  "difficulty_score": 72,
  "difficulty_label": "medium",
  "source_block_ids": ["block_id_1", "block_id_2"],
  "confidence": 0.86,
  "notes": "可作为前端基础高频题"
}
```

### 5.2.2 字段规则

- `question`：必须是明确可提问的问题。
- `answer`：可以为空，但需要标记为待补全。
- `tags`：建议 1 到 6 个。
- `difficulty_score`：0 到 100，数值越高越难。
- `difficulty_label`：`easy`、`medium`、`hard` 三档。
- `source_block_ids`：用于追溯飞书来源。
- `confidence`：0 到 1，表示 AI 对抽取结果的置信度。

### 5.2.3 难度分数标准

- `0-30`：基础概念题，适合热身。
- `31-60`：常规面试题，需要系统表达。
- `61-80`：进阶题，涉及原理、边界情况或实际经验。
- `81-100`：高难题，需要综合设计、源码理解或复杂场景分析。

### 5.2.4 导入确认

AI 抽取结果不能直接入正式题库，必须经过确认。

导入确认页面需要支持：

- 查看题目、答案、标签、难度分数。
- 修改题目和答案。
- 修改标签和难度。
- 删除不需要的题。
- 批量确认入库。
- 标记低置信度题目。

## 5.4 大模型抽题

### 5.4.1 抽题原则

抽题不是简单随机，而是由大模型根据候选题和用户状态选择。

大模型需要综合考虑：

- 用户选择的目标岗位或练习方向。
- 题库范围。
- 标签覆盖。
- 难度分布。
- 历史练习情况。
- 用户薄弱题。
- 最近是否重复出现。
- 本轮题目之间的层次关系。

### 5.4.2 抽题输入

系统传给大模型的信息包括：

```json
{
  "goal": "准备前端一面",
  "question_count": 8,
  "difficulty_preference": "medium_to_hard",
  "required_tags": ["JavaScript", "React"],
  "candidate_questions": [
    {
      "id": "q_1",
      "question": "React key 的作用是什么？",
      "tags": ["React"],
      "difficulty_score": 45,
      "last_practiced_at": "2026-05-20",
      "last_score": 62,
      "wrong_count": 2
    }
  ]
}
```

### 5.4.3 抽题输出

大模型返回：

```json
{
  "selected_questions": [
    {
      "id": "q_1",
      "reason": "React 高频基础题，用户历史得分偏低，适合作为本轮热身题"
    }
  ],
  "overall_strategy": "先用基础题热身，再进入原理和项目追问"
}
```

### 5.4.4 系统兜底

如果大模型抽题失败，系统需要使用规则兜底：

- 优先抽未练习题。
- 其次抽历史低分题。
- 控制难度分布。
- 避免最近一轮已出现的题。

## 5.5 AI 面试练习

### 5.5.1 面试会话

一次面试会话包含：

- 练习目标
- 题目列表
- 当前题目
- 用户回答
- AI 反馈
- 总结报告

### 5.5.2 单题反馈

用户回答后，AI 需要输出：

- 回答评分，0 到 100。
- 回答亮点。
- 缺失点。
- 更好的参考表达。
- 是否建议追问。
- 追问问题。

### 5.5.3 会话总结

完成一轮练习后，AI 输出：

- 总体评分。
- 表现最好的标签。
- 薄弱标签。
- 建议复习题。
- 下一轮练习建议。

## 9. 技术与运行配置

### 9.1 技术栈选择

第一版以最快开发和最少工程复杂度为优先级，采用单体全栈应用，不拆独立前后端服务。

推荐技术栈：

- 应用框架：Next.js App Router
- 开发语言：TypeScript
- UI 实现：Tailwind CSS + shadcn/ui
- 表单：React Hook Form + Zod
- 数据库：PostgreSQL
- ORM：Prisma
- 缓存和任务状态：Redis
- AI 调用：统一 OpenAI-compatible LLM Client
- 鉴权：第一版使用简单账号登录，优先 NextAuth/Auth.js 或自定义邮箱密码登录
- 部署：优先单应用部署，后续再拆 worker 或队列服务

选择原因：

- Next.js 可以同时承载页面、API Route 和服务端逻辑，适合快速做 MVP。
- Prisma 能快速定义数据库模型和迁移，便于后续迭代。
- shadcn/ui 和 Tailwind 适合快速搭建一致、可维护的后台型界面。
- Redis 只承担缓存、导入状态和限流，不在第一版引入复杂消息队列。

前端实现约定：

- 第一版直接在代码中实现界面，不依赖 Figma 设计稿。
- 组件命名围绕业务语义，例如 `QuestionCard`、`ImportPreviewTable`、`InterviewPanel`。
- 第一版优先实现桌面端体验，移动端保证可用但不做深度优化。

### 9.2 数据存储

第一版使用 PostgreSQL 作为主数据库，Redis 用于缓存 token、导入任务状态、AI 抽题上下文缓存和限流计数。
